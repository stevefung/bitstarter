Over the past few years, Android has had an incredible growth in the number of devices being used. Android currently controls about 52 percent of the mobile market share. In addition, there has lately been a rise of other, non-cellular android devices in the consumer marketplace. Products like Ouya, Panda Board, Raspberry Pi, Google TV, Samsung Galaxy Camera, and others, will soon lead to consumers having multiple Android-powered devices in use throughout their daily life. As this number increases, it would be convenient for there to be an easy way for these devices to communicate with each other, and for a user to remotely control all their devices through a simple interface.

The aim of this project is two-fold. The heart of the project is an Android service that runs in the background on any Android device. This service exposes both an internal API, as well as an external API. The external API will allow for remote commands to be sent into the system. The service will itself have some basic integrations with the OS, so that remote commands can trigger events such as launching of specific apps, turning the device on and off, changing system settings, etc. The internal API will expose an easy integration for other application developers to use with their applications, for more advanced remote command possibilities.

FAQ

Will this be available for IOS?
No. IOS contains more rigid application permissions, which make this design much more challenging. Additionally, as a closed operating system, IOS will not have the same type of potential proliferation into consumer's homes.
What are the risks involved?
There are several risks involved. First, if security permissions for Android change, that could impact the design and deliverability of such a service. Also, the usefulness of such a service will partially be determined by how other developers integrate with it, as well as how many other non-cellular devices begin to use Android in a consumer's home.
